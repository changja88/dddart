🔧 수정 모드 — 국소 변경은 풀 파이프라인을 다시 돌지 않는다

기존 구조 내 변경이면 영향받는 부분만. 리뷰는 4축 전부가 아니라 touched-layer 기계 매핑으로 줄인다(domain→ddd / infra→data / application→state / presentation→ui · ddd는 항상). 설계 변경 없는 순수 구현 수정이고 승인된 명세가 있으면 G1′도 생략. 두 경로 모두 G2 직전 백스톱 51종을 동일하게 돈다.

읽는 법 — 위에서 아래로 정상 진행. 점선 배지는 되돌림 3종류: 사용자 게이트 거부(G0/G1/G2 → 해당 Phase 재실행) · 백스톱 exit2(결정적 차단 → coordinator가 발견 성격대로 설계/coder 반송) · 서브에이전트 보고·미니게이트(coder 반송 → Model 재개봉/설계 · 미니게이트 다름 → 설계 반송).
리뷰는 항상 4축(ddd·ui·state·data) — 활성 lens를 추론하지 않는다(특히 state는 설계 중에 드러나 활성 오판이 침묵 사각이 된다 · 미활성 오판 ≫ 병렬 리뷰 1회 비용). 관심사 없는 리뷰어는 "해당 없음 + 근거 한 줄"을 의무로 낸다.
조건부·선택 단계(점선 노드) — ＊설계 경량 점검은 명세 복잡 시 coordinator 재량(생략 가능), ＊슬라이스별 감사는 슬라이스 3개 이상일 때만 + Model/View 경계 통과 시 경량 1회(판정 소유 감사 최적 시점).
홀리스틱 감사와 백스톱은 끝에 1회만 돈다(슬라이스마다 아님 — ✅ 전체 1회 박스). 백스톱은 결정적 스크립트라 코드 완성 상태에서 봐야 거짓양성이 없다. 통과(0)도 discipline-reviewer의 의미 점검(판정 소유·빈혈·이름 위장)을 면제하지 않는다.
coder는 bottom-up + analyze 베이스라인 green 래칫으로 구현한다(전용 테스트 작성 단계는 파이프라인에 없다 — FC(UI 행위 골든)는 채점 장치이지 빌드 단계가 아니다). 같은 지식 스킬이 여러 역할에서 재사용된다 — discipline-*·implementation-*는 감수자 전용이 아니라 작성자(coder)도 로드한다.
