## 2. lens 활성 규칙 — 4축 전부 항상 활성

dddjango의 "스코프에서 활성 lens를 추론·제안 → G0에서 사용자 조정" 방식을 채택하지 않는다. **ddd·ui·state·data 리뷰어 4종을 매 빌드 전부 호출한다.** 근거:

1. **활성 판단 자체가 기계 판별 불가 판단** — 특히 state(교차 BC 전파 필요 여부)는 스코프가 아니라 *설계 중에 드러나는 사실*이라 G0 시점 추론이 구조적으로 틀릴 수 있고, 클라이언트에선 사용자도 그 시점엔 모를 수 있다(서버의 "새 엔드포인트 유무"와 다름).
2. **활성 오판은 잡을 사람이 없는 사각** — 리뷰어의 안전망 규칙("명세가 다뤄야 할 것을 빠뜨리면 누락 자체를 발견으로")은 그 lens가 활성일 때만 발화한다. 파이널 리뷰의 막힘이 전부 동적 배선이었던 실측상, state 리뷰어가 꺼진 빌드가 가장 위험한 빌드다.
3. **비용 비대칭 + 절약 효과 미미** — 미활성 오판(설계 사각) ≫ 불필요 활성(병렬 리뷰 1회). 클라이언트 기능은 거의 항상 화면+상태+서버 데이터를 모두 건드려 조건 활성으로 아끼는 호출이 몇 안 된다.
4. **비결정성 제거** — 같은 기능이면 항상 같은 리뷰 구성(재현성). G0의 lens 제안·승인 절차가 사라져 Coordinator도 단순해진다.

대신 각 리뷰어에 다음 규칙을 박는다: **자기 관심사가 이 기능에 실제로 없다고 판단하면 "해당 없음 + 근거 한 줄"을 명시한다** — 부재를 침묵·생략이 아닌 검증된 판정으로 만든다. 명세가 자기 관심사를 다뤄야 하는데 빠뜨렸으면 누락 자체를 발견으로 올린다(dddjango 동일 — 항상 활성이므로 이제 항상 발화한다).

## 3. state vs data 경계 (한 주제 한 소유자)

- **data** = 데이터가 앱 바깥(서버·디스크)과 어떻게 오가는가 — 주로 `infra_layer`.
- **state** = 들어온 데이터가 앱 안에서 화면들 사이에 어떻게 살아 있는가 — 주로 VM·State·SharedState.
- 경계 판례 ① **캐싱**: hive 저장 = data, 메모리 keepAlive = state(목적이 같아도 소유자가 다름).
- 경계 판례 ② **에러**: 서버 에러가 오는 모양(Either 계약) = data, 그 에러를 VM State에 담아 표시·소비하는 방식 = state.

state 독립의 실측 근거: HaffHaff에서 refresh_notifier가 8개 BC의 VM 12개에 분포, 파이널 리뷰의 막힘 2건이 모두 동적 배선(상태 협력)이었다. 화면 구조와 상태 설계는 변경 이유가 다른 축이다.

## 5. 서버 계약(OpenAPI) 입력 설계

data lens 리뷰는 두 겹: **① 내적 정합성**(가정 계약의 명시성·완전성·규약 정합 — 출처 없이 항상 가능) / **② 외적 사실성**(실계약 대조 — 진실 출처 필요). ②의 출처를 다음으로 설계한다.

- **고정 위치·고정 키**: 대상 프로젝트의 `.dddart/config.json` → `"openapi_url"`. 사람이 미리 직접 써놔도 인식한다.
- **우선순위**: 커맨드 인자 > config. 인자로 주면 config에 저장/갱신 → 이후 실행은 다시 묻지 않고 "서버 계약 출처: <url> (저장된 설정)" 한 줄 보고.
- **계약 스냅샷**: 빌드 시점에 관련 엔드포인트만 잘라 `.dddart/<생성일>-<기능>/server-contract.json`으로 동결. *왜* — architect·data 리뷰어·coder가 같은 계약을 보고(결정성, 빌드 중 서버 배포에 안 흔들림), 전문 주입 없이 컨텍스트를 아낀다.
- **읽기 주체 규칙**: config·라이브 URL은 **Coordinator만** 읽는다(출처 해소는 한 소유자). 하위 에이전트는 **스냅샷 파일만 본다(라이브 URL·config 접근 금지)** — dddjango에서 하위가 스코프 메모 아닌 명세만 읽는 규율과 동형.
- **출처 폴백 순서**: OpenAPI(기계 대조) → 기존 DataSource 패턴(같은 서버의 살아 있는 계약 증거) → 가정 계약을 명세에 박고 **G1 배너 사용자 확인 항목으로 승격**(dddjango의 "기본 미적용 + 배너 override" 패턴). 출처 없이 사실성을 판정한 것처럼 보고하지 않는다.
- **안내 3곳**: 커맨드 `argument-hint`(`"[기능 설명] (선택: OpenAPI 주소 — 서버 계약 대조가 정확해집니다)"`) · G0에서 출처 없을 때 1회 안내(답하면 저장되어 프로젝트당 1회) · README 시작하기 절.

## 7. §10-1 본설계로 넘기는 것

이 문서는 *구성*만 확정한다. 다음은 본설계 소관: phase 구조(G0/G1/G2 절차·배너 형식·수정 모드), 진행 가시성(트래커·한 줄 상태), 산출물 폴더 규약(`.dddart/` 상세), 에이전트별 스킬 주입 슬라이싱(규약 어느 절을 누구에게 — 기계 판별 불가 17종의 리뷰어 체크리스트 편입 포함), G2 행위 체크리스트 형식, coder의 슬라이스 내 작업 순서(계층 순서 등).
