## 종합 판정

게이트·소유권·표면화 골격(dddjango 이식부)과 구조·import 백스톱 체계는 5렌즈 공격을 견뎠고, **리뷰어 4종 항상 활성은 비용 검증도 통과**했다(리뷰어 4종 정적 비용은 전체의 ~7–12%, 침묵 사각 1회의 기대 비용이 압도 — 단 풀 빌드 한정). 그러나:

1. **파이프라인의 존재 이유인 "판정의 자리"(빈혈 차단)에 집행자가 없다** — 신규 기능의 판정 1곳(1-카피)은 규칙·리뷰어·감사자·백스톱 전 단계가 못 잡고, 유일한 규칙(VM 2곳 복제 강등)은 발화 조건을 관측할 에이전트가 없다.
2. **테스트 제거의 숨은 청구서 3장** — dddjango가 pytest에 무임승차시킨 ① 완성 판정 베이스라인 ② 반송 후 재진입 앵커 ③ 진행 상태 기록이 대체 없이 사라졌다. 여기에 Flutter 고유 현실(codegen·브라운필드 analyze 소음·빌드 툴체인)이 미반영 — **현 문서로 첫 브라운필드 실행 시 Phase 2 래칫에서 멈출 확률이 높다.**
3. **dddart가 새로 발명한 메커니즘 3종(계약 스냅샷·디자인 동결·git 스냅샷)은 "무엇을"만 있고 "어느 도구로 어떻게"가 비어 있다.**
4. **"기계 도출(LLM 재량 없음)" 주장이 자기 절 안의 비기계 요소 3개로 무너진다** — 안 1 채택의 1번 근거였으므로 치명적.

전부 §10-2~4 착수 전 본설계 수정으로 해소 가능 — 설계 방향 자체를 뒤집는 발견은 없음.

## A. Blocker 주제 11 (렌즈 간 병합·중복 제거)

### A1. 빈혈 1-카피 집행 공백 (집행력 B1 — 단독 최대 발견)
신규 기능의 모든 판정은 소비처가 1곳이므로 "단일 화면 표시 로직 → VM 소유"로 명세해도 문면상 합법. ddd 리뷰어 점검이 "2곳 복제 강등"으로 약화 정의됐고, dddjango discipline-reviewer의 빈혈 직격("새 판정 도입에 domain 판정 코드 0 = blocker")의 dddart 대응물이 없으며, 백스톱 39종·17종 목록 어디에도 판정 소유가 없다. 강등 규칙 자체도 — 리뷰어는 코드를 안 보고, architect 조사 의무에 기존 판정 인벤토리가 없고, coder에 grep 절차가 없어 — **발화 조건을 관측할 자가 없다**. 우회 변종: ui_extension getter·State getter·view 분기는 "VM 2곳" 문면을 영원히 비껴간다.
**처방**: ① ddd lens·architecture-ddd에 양성 규칙 — "행위 목록의 모든 수치·비교·자격 판정은 명세에서 소유자를 항목별 라벨링, 도메인 어휘 판정은 1곳째부터 domain 기본(VM 주장 시 *왜* 요구)" ② discipline-reviewer에 C형 직격 이식 ③ 강등 규칙 문면을 "Model 밖 2곳(VM·view·section·ui_extension·State getter)"으로 확장 ④ coder에 기존 BC 수정 시 같은 판정 grep 절차.

### A2. §8 스킬 귀속 ↔ §9 검증자 배정의 구조적 모순 (집행력 B2)
discipline-reviewer가 §9 표에서 최소 9개 판별의 검증자인데, 그 판별의 원문(§3.2·§3.3·§3.5·§3.6·§6)은 전부 architecture-*(설계 스킬)로 귀속 — discipline-reviewer 코퍼스(cleancode+houserules)에 없다.
**처방**: 17종 판별 절차를 별도 공유 reference로 빼고 1차 결정자와 검증자 양쪽이 같은 파일을 적재(§10-3·§10-4 요구사항에 명문화).

### A3. View 슬라이스의 Model 수정 무규정 — 빈혈 역형태 경로 (집행력 B3)
State 필드 부족을 View coder가 발견했을 때 반송 의무 조문이 이 경우를 커버하지 않음 → view 우회 계산(빈혈, 행위 대조도 통과) 또는 무감사 Model 수정(경계 감사 전제 침묵 파괴).
**처방**: 반송 열거에 추가 — "슬라이스 계층 밖 파일 수정 필요 시(State 부족 포함) 수정·우회 모두 금지·보고. View 슬라이스에서 Model 파일이 변경되면 경계 감사 1회 재실행."

### A4. "기계 도출" 자기모순 (정합 B1 + 절차 I1·I3 + 플러그인 I7 — 3렌즈 수렴)
① 줄 수 임계(~1.2k)는 코드 작성 전 존재하지 않음(추정=재량) ② "~8" 등 물결표의 비결정 ③ tracer 트리거 "위험 행위"가 무정의 — 18번째 기계 판별 불가 규칙이며 §9 배정에도 없음 ④ hive 어댑터 파일이 Model(infra)과 View(배선)에 이중 귀속.
**처방**: 도출 규칙은 파일 수 정수로 전 구간 정의(예: ≤7 축퇴 / 8 유지 / ≥9 세분), 줄 수는 coder 호출 내 사후 트리거로 강등, tracer는 기계 플래그("가정 계약 G1 승격 케이스")만 — '위험 행위'는 architect의 명세 결정 항목으로 승격(§9 표에 행 추가), hive 어댑터 파일 생성=Model·root_initializer 조립 1줄=View 배선으로 분리 서술.

### A5. 브라운필드 analyze green 미정의 (절차 B1)
기존 앱의 경고 수백 개·기존 error가 있으면 "green이어야 다음으로"가 영원히 불충족이거나 coder 자가 판정(비결정 부활).
**처방**: Phase 2 진입 시 analyze 베이스라인 1회 캡처(산출물 폴더 기록), green = "베이스라인 대비 신규 이슈 0". 파일별 green = "touched 파일에 error 0".

### A6. codegen(build_runner) 단계 부재 (절차 B2)
@riverpod·@freezed 작성 직후 `*.g.dart` 부재로 analyze는 무조건 red — 래칫이 구조적으로 깨짐. dddjango "러너 준비"의 대체가 없다.
**처방**: 러너 준비 동형의 "codegen 규약" 신설 — analyze 전 build_runner 실행 시점·주체, 미설치 시 버전-핀 설치 규율, codegen 오류와 analyze 오류의 구분 보고.

### A7. git 스냅샷 실체 미정의 (절차 B3 + 플러그인 B3 — 수렴)
비git 프로젝트 폴백 없음, dirty tree에서 revert가 사용자 WIP 파괴 가능, ref 기록 위치가 대화 컨텍스트뿐(세션 사멸 시 소실).
**처방**: G0 전제조건 검사(git 여부·청결 — 더러우면 배너 표면화), 슬라이스 green마다 커밋(§2 기본 커밋과 정합), 스냅샷 ref는 산출물 폴더 상태 파일에 기록, revert는 "이번 파이프라인 커밋 이후만".

### A8. 반송 후 재진입 절차 부재 (절차 B4)
구현 중 설계 반송 → 명세 파일 목록 변경 → 완료된 Model 슬라이스의 처분(재도출? stale 집행?)이 미정의. 시뮬레이션 자신이 "가로 계열의 반송 의미론 붕괴"를 경고했는데 그 방지 절차가 이식되지 않음.
**처방**: architect 반송 재호출 산출에 "변경 파일 diff" 요구 → Coordinator가 diff 기준 슬라이스 재도출·영향 슬라이스만 재개봉(수정 의미론으로 coder 입력 정의).

### A9. 수정 모드 G1' 생략 경로의 슬라이스 도출 입력 부재 (절차 B5)
순수 구현 수정은 명세 갱신이 없는데 도출은 "명세의 파일 목록에서" — 합성 불가.
**처방**: G0 영향 파일 목록(스코프 메모 산출)을 도출 입력으로 승격 + G0 배너 승인 항목화.

### A10. 신규 메커니즘 2종의 도구 공백 (플러그인 B1·B2)
① server-contract 추출: 네트워크 도구 미지정, openapi.json의 `$ref` 전이 참조를 LLM 절단이 깨뜨림(dangling ref) → **결정적 스크립트** `extract-contract.py`(paths 선별+$ref 폐쇄 추출)를 플러그인 scripts/에 두고 Bash 실행. ② design-ref 동결: Write는 텍스트 전용이라 Coordinator가 이미지를 저장 못 함, Stitch는 표준 MCP 부재 → **수동 폴백(사용자 내보내기 → Bash cp)을 1급 기본 경로로** 뒤집고 MCP는 "이미지 파일 경로를 만들 수 있는 서버"일 때만.

### A11. 트리비얼 하한 부재 — 우회 학습이 drift의 정문 (비용 1)
"라벨 문구 수정"도 최소 ~10–20분·개입 2회·~100K+ 토큰(맨 Claude 대비 20–30배). 우회를 학습하면 그 경로엔 백스톱조차 없다.
**처방(사용자 결정 D1)**: G0 삼분류(신규/수정/**트리비얼**) — 트리비얼 판별은 기계적(신규 파일 0+비구조 diff), 경로 = 직접 편집+touched 백스톱+analyze만(에이전트·빌드·G2 생략), 판정 자체는 사용자 승인 1회.

## B. 상충 2건 — 사용자 결정 필요

### D2. 수정 모드의 리뷰어: 4종 전부(현 설계) vs touched-layer 기계 매핑(비용 렌즈)
비용 렌즈의 반박: 4축 항상의 폐지 근거는 "G0 시점 *추론*의 불확실성"인데, **수정 모드는 touched 파일이 이미 알려져 있고 계층 폴더가 lens를 기계 매핑한다**(domain→ddd, infra→data, VM/State→state, presentation→ui) — 추론이 아니라 기계 신호라 침묵 사각 논거가 소멸. 낭비 ~35–50K/수정 빌드. **권고안**: 수정 모드 한정 touched-layer 기계 매핑 채택(ddd는 항상 유지), 또는 최소한 2단계 실측 지표 편입.

### D3. 수정 모드의 discipline 감사: 홀리스틱 필수(집행력 I2 — 빈혈이 가장 누적되는 모드) vs 비앵커(비용 7 — dddjango 동형)
**권고안(절충)**: 트리비얼 채널(D1)이 라벨급을 흡수하는 전제에서, 수정 모드는 "touched 파일 한정 경량 감사 1회"로 — 빈혈 감시는 유지하되 비용은 홀리스틱의 일부.

## C. Important (요약 — 처방은 반영 시 상세화)

1. **push service navigator 백스톱 부재**(집I1+정합 — 수렴): §3.6 "백스톱 검사 대상" 선언이 39종에 없고 채널 ③(navigator)이 합법화 → §10-2에 "`service/**`에서 `_navigator` import·`.go(` 금지" 추가, §3.7 주석 정밀화("VM→navigator만").
2. **§3.7이 application→design_system 전면 허용**(집I5): VM의 시각 토큰 매핑·BuildContext 보유가 백스톱 사각 → §10-2에 "view_model·shared_state·use_case에서 design_system import 금지"·"application_layer에서 BuildContext·material import 금지" 추가 + §3.7 단서.
3. **Model 표시 문구 역류**(집I3): "Model은 에러를 타입/코드로, 표시 문구는 VM·ui_extension이" — §10-5 ① 작성 시 포함 + data 리뷰어 점검(Left 페이로드 형태).
4. **17종 배정 모순 3건**(정합): "맥락" 검증자 ddd→ui, 과거형 사건명 1차 coder→architect(파일명은 명세 소유), "export된 view"는 M5 해소로 백스톱 이관(표에서 삭제 — 실질 16종).
5. **agent-comp 잔재 2건**(정합 I4): "행위 목록을 슬라이스로 변환"·"VM·view는 같이 만들어야"(2분할과 모순) — 개정 일자 부기 수정.
6. **감사 임계 4개↔≥3 모순**(정합 I6): dddjango 원문은 ≥3 — 통일.
7. **축퇴 ~7 vs 수정 임계 ~6 모순 + 신규/수정 단위 불일치**(정합 I7+절차 N5): A4의 정수 정의로 함께 해소.
8. **모드 판별 내적 모순**(절차 I6): "신규 파일이면 풀" vs "수정 모드 대부분 신규+수정 ≤7" — 모드 기준을 구조 단위로 재정의(신규 화면·애그리거트·BC·라우트 = 풀 / 기존 구조 내 = 수정).
9. **중간 눈 확인 미정식**(절차 I4): 재량→기계 발동(가정 계약 = 항상), 미니 게이트로 정식화(배너·3분기 반송).
10. **flutter build 환경 의존**(플I5+절I5+비용6 — 수렴): 타깃 결정 절차, 환경 실패=미실행 보고(반송 아님), 수정 모드는 조건부 빌드(codegen·에셋·네이티브·pubspec touched 시만), G2 배너에 "사용자가 `flutter run`으로 실행" 명시.
