# 클린 코드 규율

## 언제 쓰나

코드를 작성·리뷰·리팩토링하며 "이해 가능한가, 응집적인가, 명시적인가, 동작을 보존하며 바꿀 수 있는가"를 물을 때 로드한다. 전문(90KB)을 읽지 말고 아래 라우팅 표로 필요한 절만 부분 적재한다. 경계:

- 도메인 모델링·판정 소유·빈혈 판별 → `architecture-ddd`
- 파일트리·폴더·명명 **사실** → `discipline-houserules`
- VM·State 동작 계약 → `architecture-state`
- Either·실패 처리 계약 → `architecture-data`
- Dart 관용구·표기법 상세 → `implementation-dart`

## 핵심 운영 원칙

- 이름은 의도를 드러낸다 — 검색 가능·한 개념 한 단어, 핵심 의미를 앞에 (§2)
- 함수는 작게, 한 가지 일만, 한 단계의 추상화 — 플래그 인수는 함수 분리 신호 (§3)
- 주석은 "왜"를 적는 보완 수단 — 코드로 표현 가능한 것을 주석으로 때우지 않는다 (§4)
- 내부 표현과 불변식은 객체 안에 — 묻지 말고 시켜라, getter 체인으로 파고들지 않는다 (§6·§8)
- 깊은 모듈: 좁은 인터페이스 뒤에 큰 기능 — 얕은 통과 계층을 늘리지 않는다 (§7)
- 오류는 일찍, 명시적으로 — 침묵 폴백·광범위 catch 금지 (§12)
- DRY는 지식 단위로 판단한다 — 같은 지식의 중복만 모으고, 우연히 닮은 코드(다른 도메인 지식)는 합치지 않는다 (§13, 표기 반복의 경계는 §18)
- 리팩토링은 동작 보존이 전제 — 스멜 카탈로그·기법은 §15, dddart의 안전망(analyze green 래칫·백스톱·G2 행위 대조 + 행위 검증 테스트[coder 산출·green=flutter test])은 §16 머리 단서
- 레거시 변경은 Sprout/Wrap — 기존 코드를 최소로 건드린다("기존 코드 수정 불요구"와 한 방향) (§16)
- **반복>상속(dddart 결정)**: base VM·패턴 추상화 금지 — 지식의 중복은 도메인으로 모으고, 표기의 반복(listen 3줄·위임 1줄)은 정식 예제를 그대로 반복한다 (§18)
